Agent Control Audit

Watching an agent get audited — live, in its own terminal. This walkthrough follows one real run against a customer-facing financial agent, from first command to final decision.

Target: financial_agent Framework: OpenAI Agents SDK Risk tier: Critical
THE CAPABILITY

One question, asked rigorously

Given an agent someone already built — what guardrails, evals, governance evidence, and operational controls are missing before anyone should trust it?

Why it exists

  • Most agent demos show the happy path only
  • Real governance teams ask harder questions: can it move money, get bypassed via prompt injection, mishandle abuse, ship without evals?
  • This turns those questions into a repeatable audit with evidence — not a vague "review for safety" prompt

The core idea

  • Required controls are derived first — from harm surface, autonomy, architecture, domain
  • That required set is then compared against what's actually source-visible
  • So the tool can't quietly skip a control just because it didn't happen to find one

01 — MEET THE TARGET

A customer-facing agent that moves real money

What it does

  • Handles customer requests and financial advice via RAG retrieval
  • Hands off to a portfolio-specialist agent for complex cases
  • Calls transfer_funds directly against live customer accounts

Why it's "critical" tier

  • Autonomous, side-effecting money movement — no human in the loop by default
  • Multi-agent handoff widens the surface for bad instructions to slip through
  • Regulated domain: financial advice + fund transfer both carry compliance weight
02 — HOW THE AUDIT THINKS

Not "review for safety." Derive, then compare.

The audit doesn't ask an LLM to vaguely judge the agent. It derives a required control set from the agent's harm surface, autonomy, and risk tier — then checks that set against what's actually visible in source.

Required set

34 controls across 14 requirement groups, FIN‑001 through FIN‑014, specific to money‑moving agents.

Source evidence

Static pass reads the actual agent code — no assumptions about intent, no self-reported compliance.

Decision

Any blocker-severity gap in the required set is an automatic BLOCK, regardless of what else is present.

07 — THE REMEDIATION PLAN

Wave 1 — must fix before anything else

1
Tool Authorization
Bind a caller-identity / account-ownership check inside transfer_funds; fail closed without auth context.
2
Human Approval Gate
Add an interrupt/approval pause that surfaces full transfer details before execution.
3
Limits & Kill Switch
Add per-transaction cap, aggregate cap, recipient allowlist, and an instant kill switch.
4
Semantic Argument Validation
Extend schema validation with ownership checks, idempotency keys, and type constraints.
5
Durable Audit Logging
Replace print() with structured logs to a durable sink, capturing outcome and guardrail trips.

09 — IS THE DATA USABLE

Dataset Readiness

How

  • Checks the golden dataset against schema validity, case counts, and machine-checkability
  • Flags missing metadata and thin adversarial/benign coverage before any eval runs

Why

  • Eval results are meaningless if the underlying test data isn't trustworthy
  • This is the gate that decides whether pass/fail numbers count as evidence — or just noise
Status: eval_ready
10 — PROVING THE HARNESS ITSELF

Three checkpoints before touching a real agent

Before DeepEval (Confident AI's open-source eval framework) ever points at a real agent, the harness proves itself against a safe mock first.

1
Mock agent + seed dataset
Isolates whether the judge/rubric pipeline itself works — separate from whether any real agent behaves.
2
Mock agent + real/domain dataset
Exercises the actual dataset's structure and assertions end-to-end, still against the safe mock.
3
Real agent + real dataset, over HTTP
Only run once 1 and 2 are green — the first time the real agent is actually touched.
11 — DID IT PASS

Eval Summary

Runs the DeepEval suite in two layers — proof the agent behaves at runtime, not just that the code looks clean in source.

Layer 1 · Hard gates

Deterministic checks — tool calls, approval requests, blocked flags, citations. A good judge score can never override a failed gate.

Layer 2 · GEval judge

Scores 0.0–1.0 against a 0.7 pass threshold. Only runs after every hard gate has already passed.
